# Fixture-as-oracle audit — 2026-06-14

Discharges the consequence in
[`../decisions/2026-06-14-cue-compatibility-target.md`](../decisions/2026-06-14-cue-compatibility-target.md):
`testdata/cue/*.expected` was generated by cross-checking the official `cue eval` binary,
so it could encode v0.15 binary bugs rather than the intended semantics. This audit
checked all 82 fixture pairs against the **spec**, not the binary.

## Method

One read-only pass, fanned to parallel subagents over four semantic domains, each judging
`.expected` against CUE v0.15 spec semantics (cuelang.org + RE2), flagging only
high-confidence wrong outputs: arithmetic/expressions (16), structs/lists/manifest (28),
disjunctions/bounds/refs/cycles (16), regex/patterns/builtins (23).

## Result

**No high-confidence suspect fixtures** — including the spots most prone to
reference-binary bugs: apd decimal arithmetic (`0.1 + 0.2 = 0.3` exact, `1/3` to 34
digits); `div`/`mod` Euclidean vs `quo`/`rem` truncating (signs correct); structural
cycles (`x: x & >=0` → `>=0`, pure ref cycles → `_`); disjunction defaults +
subsumed-disjunct elimination (`number | 1` → `number`); closedness rejections; RE2
matches.

Kue's deliberate output conventions (inline `field: _|_` for conflicts vs the binary's
top-level hard error; hidden fields shown as with `cue eval -a`) are presentation, applied
uniformly — not semantic divergence.

**Lone open item** (low confidence, already documented): `number_literals` renders
`1.25e3` as `1.25e+3` rather than `1250` — a documented boundary in
[`../spec/compat-assumptions.md`](../spec/compat-assumptions.md)
§Numeric literals. Not a bug; pending a future canonicalization slice.

## Takeaway

The suite is trustworthy as an oracle for the implemented surface; TDD slices can lean on
it without inheriting reference-binary bugs. No fixture changes made.
